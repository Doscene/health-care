# 家庭慢病健康管理应用 — 开发计划

## 一、项目概况

| 维度 | 说明 |
|------|------|
| 项目名称 | 家庭慢病健康管理应用 |
| 产品定位 | 高血压+糖尿病家庭的日常健康管理工具 |
| 目标用户 | 老人（患者）、年轻患者、子女（照护者） |
| MVP范围 | 纯移动端App（Android原生），无IoT/医生端/视频通话 |
| 核心能力 | 指标记录、智能用药、家庭协作、分级预警、饮食助手 |
| AI能力 | 语音记录 + 基础OCR药盒识别（多厂商兼容） |
| 总工期 | **14周** |
| 团队配置 | Android 2人 + 后端 2人 + 测试 1人 = 5人 |

## 十一、风险识别与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| OCR识别准确率不达标 | 中 | 高 | 多厂商Fallback；提供手动录入兜底；Phase 0即可开始OCR联调 |
| 语音识别方言/噪音问题 | 中 | 中 | MVP只支持普通话；老人端保留大按钮手动录入 |
| 医疗合规风险 | 低 | 高 | 所有页脚标注"仅供参考，不构成医疗建议"；不提供诊断功能 |
| 推送到达率低（厂商限制） | 中 | 中 | 多通道（JPush + App内消息）；关键提醒用短信兜底 |
| 隐私/数据安全审计 | 低 | 高 | VisibilityGuard 代码级控制；手机号加密存储；敏感数据脱敏 |
| Phase交织导致返工 | 中 | 中 | 严格按Phase交付；API契约先行（先定接口再并行开发） |
| 人力不足/请假 | 中 | 高 | 每个Phase核心模块指定主+备负责人；P0任务不过度并行 |

## 十三、开发规范

### 13.1 代码管理
- **分支策略**：`main`（保护）← `develop` ← `feature/phaseX-module-name`
- **提交规范**：`feat:` / `fix:` / `refactor:` / `docs:` / `test:`
- **Code Review**：至少1人Review后合并到develop

### 13.2 接口约定
- API先行：每个Phase开始前，后端先出Swagger接口文档
- Android按接口Mock数据并行开发，后端就绪后联调
- 统一响应格式：`{ code: number, data: T, message: string }`

### 13.3 测试要求
- 单元测试：核心业务逻辑（预警引擎、OCR兼容层、隐私守卫）覆盖率 > 60%
- 接口测试：每个接口至少1个正向+1个异常用例
- UI测试：关键用户旅程（记录→保存→查看）自动化测试

### 13.4 文档产出
- 每个Phase结束产出：`docs/phaseX-验收报告.md`
- 接口文档：Swagger自动生成，随代码更新
